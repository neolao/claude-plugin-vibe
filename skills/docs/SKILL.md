# /vibe:docs ‚Äî Project Documentation Maintainer

Generate or refresh the project's user-facing documentation: managed sections in `README.md` and generated files in `docs/`. Documentation is derived from real project state (manifest, CHANGELOG, `.vibe/`, code) ‚Äî never invented.

Two golden rules:
1. **Never touch anything outside the managed markers in README.md.** Everything else belongs to the user.
2. **Write for the end user** ‚Äî no file names, function names, or internal module names (same rule as CHANGELOG entries).

## Step 1 ‚Äî Read the sources

- `README.md` if it exists ‚Äî locate managed sections (see Step 3) and note the manual content around them
- The project manifest(s) ‚Äî name, description, install/run/test scripts
- `CHANGELOG.md` if it exists ‚Äî versions and entries
- `.vibe/index.md`, `.vibe/modules/*.md`, `.vibe/glossary.md` if they exist ‚Äî architecture and terminology
- `.env.example` (or equivalent) if it exists ‚Äî configuration surface
- `CLAUDE.md` ‚Äî stack and project type
- `docs/` ‚Äî list every file currently present (generated by this skill or hand-written); each one must end up referenced in README (see Step 4)
- `$ARGUMENTS`: if it contains `--full`, force full regeneration of `docs/` files

## Step 2 ‚Äî Create task list

Create tasks using TaskCreate, chained with `addBlockedBy`. **Keep subject names short (‚â§ 30 chars)** ‚Äî they appear in the status line.

```
Update README sections   ‚Üê no dependency
Update docs/             ‚Üê blockedBy "Update README sections"
```

## Step 3 ‚Äî Update README managed sections

Mark the `Update README sections` task `in_progress`.

Managed sections are delimited by HTML markers. Only the content **between** a `begin`/`end` pair may be rewritten:

```markdown
<!-- vibe:begin:features -->
...generated content...
<!-- vibe:end:features -->
```

| Section | Markers | Content derived from |
|---|---|---|
| Features | `vibe:begin:features` / `vibe:end:features` | `CHANGELOG.md` ‚Äî user-facing capabilities across all versions + [Unreleased]; deduplicated, phrased as a benefit list, not a changelog copy |
| Installation | `vibe:begin:install` / `vibe:end:install` | Manifest ‚Äî prerequisites (language/runtime version), the actual install command, a quick way to verify the install worked, and update/uninstall commands when the install method provides them |
| Usage | `vibe:begin:usage` / `vibe:end:usage` | Manifest scripts + real entry points ‚Äî every user-facing command with realistic examples verified against the actual code |
| Documentation | `vibe:begin:docs-index` / `vibe:end:docs-index` | Every file present in `docs/` ‚Äî one line per file: relative link + one-sentence description. This section is the index/summary of `docs/`: it must list ALL files in that folder, whether generated by this skill or hand-written. It is populated last, in Step 4, so it reflects the final state of `docs/` |

Cases:

- **README exists and contains managed sections:** rewrite only the inside of each marker pair. Do NOT modify a single character outside the markers ‚Äî not even whitespace or heading levels.
- **README exists but has no managed sections:** do not insert them silently into a hand-written file. List the sections that could be managed and ask the user which ones to add (and where: end of file by default). Only add the pairs the user approves.
- **README does not exist:** create it with a minimal skeleton ‚Äî project title, one-paragraph description (from manifest + CLAUDE.md), then the four managed sections.

Leave `vibe:begin:docs-index` / `vibe:end:docs-index` empty for now if `docs/` doesn't exist yet ‚Äî it is filled in Step 4.

Mark the task `completed`.

## Step 4 ‚Äî Update docs/

Mark the `Update docs/` task `in_progress`.

`docs/` files are **fully generated** ‚Äî each starts with this banner:

```markdown
> Generated by /vibe:docs ‚Äî manual edits will be overwritten; use README for hand-written content.
```

| File | Content | Create only if |
|---|---|---|
| `docs/architecture.md` | Public overview derived from `.vibe/index.md` + `.vibe/modules/` ‚Äî the big picture for an external reader: what the main parts are, how data flows. Reformulated in user-facing language; never a raw copy of `.vibe/` content | `.vibe/` exists and has ‚â• 2 modules |
| `docs/configuration.md` | Environment variables and options ‚Äî name, purpose, default, required or not ‚Äî derived from `.env.example` and how the code reads them | the project has configuration (env vars, config file) |

Rules:
- Never create an empty or near-empty file ‚Äî skip a file if there is no substance for it.
- **Without `--full`**: incremental ‚Äî regenerate a file only if its sources changed (compare against the current file content; if nothing meaningful changed, leave it untouched).
- **With `--full`**: regenerate every applicable file from scratch.
- If a `docs/` file exists that was NOT generated by this command (no banner): leave it alone and mention it in the report.

### Update the README documentation index

Once `docs/` reflects its final state (files created, updated, left untouched, or unchanged), rewrite the `vibe:begin:docs-index` / `vibe:end:docs-index` section in `README.md` so it lists **every** file currently in `docs/`:

- One line per file: a relative link (`[docs/architecture.md](docs/architecture.md)`) + a one-sentence, user-facing description of what it contains.
- Include hand-written files (no banner) as well as generated ones ‚Äî the README must stay a complete index of `docs/`, not just of what this skill produced.
- Remove entries for files that no longer exist in `docs/`.
- If `docs/` is empty or absent, leave the section empty (or a single "No additional documentation yet." line) rather than deleting the markers.
- Keep the list sorted the same way each run (e.g. alphabetical by path) so diffs stay minimal.

Mark the task `completed`.

## Step 5 ‚Äî Report

- README: which managed sections were updated / unchanged / absent, including the Documentation index
- docs/: files created, updated, unchanged, or skipped (with reason)
- Flag any `docs/` file that is not (yet) referenced in the Documentation index ‚Äî this should not happen, but call it out if seen
- Do NOT print file contents unless asked
- Do NOT commit ‚Äî the caller (`/vibe:feature`, `/vibe:fix`, `/vibe:release`) or the user handles commits
