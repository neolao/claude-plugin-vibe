# /vibe:docs — Project Documentation Maintainer

Generate or refresh the project's user-facing documentation: managed sections in `README.md` and generated files in `docs/`. Documentation is derived from real project state (manifest, CHANGELOG, `.vibe/`, code) — never invented.

Two golden rules:
1. **Never touch anything outside the managed markers in README.md.** Everything else belongs to the user.
2. **Write for the end user** — no file names, function names, or internal module names (same rule as CHANGELOG entries).

## Step 1 — Read the sources

- `README.md` if it exists — locate managed sections (see Step 3) and note the manual content around them
- The project manifest(s) — name, description, install/run/test scripts
- `CHANGELOG.md` if it exists — versions and entries
- `.vibe/index.md`, `.vibe/modules/*.md`, `.vibe/glossary.md` if they exist — architecture and terminology
- `.env.example` (or equivalent) if it exists — configuration surface
- `CLAUDE.md` — stack and project type
- `$ARGUMENTS`: if it contains `--full`, force full regeneration of `docs/` files

## Step 2 — Create task list

Create tasks using TaskCreate, chained with `addBlockedBy`. **Keep subject names short (≤ 30 chars)** — they appear in the status line.

```
Update README sections   ← no dependency
Update docs/             ← blockedBy "Update README sections"
```

## Step 3 — Update README managed sections

Mark the `Update README sections` task `in_progress`.

Managed sections are delimited by HTML markers. Only the content **between** a `begin`/`end` pair may be rewritten:

```markdown
<!-- vibe:begin:features -->
...generated content...
<!-- vibe:end:features -->
```

| Section | Markers | Content derived from |
|---|---|---|
| Features | `vibe:begin:features` / `vibe:end:features` | `CHANGELOG.md` — user-facing capabilities across all versions + [Unreleased]; deduplicated, phrased as a benefit list, not a changelog copy |
| Installation | `vibe:begin:install` / `vibe:end:install` | Manifest — prerequisites (language/runtime version) + the actual install command |
| Usage | `vibe:begin:usage` / `vibe:end:usage` | Manifest scripts + real entry points — main commands with realistic examples verified against the actual code |

Cases:

- **README exists and contains managed sections:** rewrite only the inside of each marker pair. Do NOT modify a single character outside the markers — not even whitespace or heading levels.
- **README exists but has no managed sections:** do not insert them silently into a hand-written file. List the sections that could be managed and ask the user which ones to add (and where: end of file by default). Only add the pairs the user approves.
- **README does not exist:** create it with a minimal skeleton — project title, one-paragraph description (from manifest + CLAUDE.md), then the three managed sections.

Mark the task `completed`.

## Step 4 — Update docs/

Mark the `Update docs/` task `in_progress`.

`docs/` files are **fully generated** — each starts with this banner:

```markdown
> Generated by /vibe:docs — manual edits will be overwritten; use README for hand-written content.
```

| File | Content | Create only if |
|---|---|---|
| `docs/architecture.md` | Public overview derived from `.vibe/index.md` + `.vibe/modules/` — the big picture for an external reader: what the main parts are, how data flows. Reformulated in user-facing language; never a raw copy of `.vibe/` content | `.vibe/` exists and has ≥ 2 modules |
| `docs/configuration.md` | Environment variables and options — name, purpose, default, required or not — derived from `.env.example` and how the code reads them | the project has configuration (env vars, config file) |

Rules:
- Never create an empty or near-empty file — skip a file if there is no substance for it.
- **Without `--full`**: incremental — regenerate a file only if its sources changed (compare against the current file content; if nothing meaningful changed, leave it untouched).
- **With `--full`**: regenerate every applicable file from scratch.
- If a `docs/` file exists that was NOT generated by this command (no banner): leave it alone and mention it in the report.

Mark the task `completed`.

## Step 5 — Report

- README: which managed sections were updated / unchanged / absent
- docs/: files created, updated, unchanged, or skipped (with reason)
- Do NOT print file contents unless asked
- Do NOT commit — the caller (`/vibe:feature`, `/vibe:fix`, `/vibe:release`) or the user handles commits
