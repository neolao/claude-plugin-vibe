# /vibe:docs — Project Documentation Maintainer

Generate or refresh the project's documentation: end-user-facing managed sections in `README.md` and developer-facing generated files in `docs/`. Documentation is derived from real project state (manifest, CHANGELOG, `.vibe/`, code) — never invented.

Two golden rules:
1. **Never touch anything outside the managed markers in README.md.** Everything else belongs to the user.
2. **Two audiences.** README managed sections are written **for the end user** — no file names, function names, or internal module names (same rule as CHANGELOG entries). `docs/` files are written **for developers** — the goal is that a new developer understands the project, so real module names, file paths, and commands are welcome, as long as they come from the actual code, never invented.

## Step 1 — Read the sources

- `README.md` if it exists — locate managed sections (see Step 3) and note the manual content around them
- The project manifest(s) — name, description, install/run/test scripts
- `CHANGELOG.md` if it exists — versions and entries
- `.vibe/index.md`, `.vibe/modules/*.md`, `.vibe/glossary.md` if they exist — architecture and terminology
- `.env.example` (or equivalent) if it exists — configuration surface
- `CLAUDE.md` — stack and project type
- The code itself — walk the real structure (tree, entry points, main modules) to identify every aspect a new developer would need explained; `.vibe/` and the manifests help, but they are not the only source
- `docs/` — list every file currently present (generated by this skill or hand-written); each one must end up referenced in README (see Step 4)
- `$ARGUMENTS`: if it contains `--full`, force full regeneration of `docs/` files

## Step 2 — Create task list

Invoke the `vibe:tasks` skill (Skill tool) to create the tasks below, chained with `blockedBy`. **Keep subject names short (≤ 30 chars)** — they appear in the status line. `vibe:tasks` creates the tasks via `TaskCreate`, or falls back to a scratchpad checklist if that tool is unavailable — either way, its instructions then govern how every later "Mark the task ... completed" instruction in this skill is carried out.

Pass these as `$ARGUMENTS`:

```
Update README sections   ← no dependency
Update docs/             ← blockedBy "Update README sections"
```

## Step 3 — Update README managed sections

Mark the `Update README sections` task `in_progress`.

Managed sections are delimited by HTML markers. Only the content **between** a `begin`/`end` pair may be rewritten:

```markdown
<!-- vibe:begin:features -->
...generated content...
<!-- vibe:end:features -->
```

| Section | Markers | Content derived from |
|---|---|---|
| Features | `vibe:begin:features` / `vibe:end:features` | `CHANGELOG.md` — user-facing capabilities across all versions + [Unreleased]; deduplicated, phrased as a benefit list, not a changelog copy |
| Installation | `vibe:begin:install` / `vibe:end:install` | Manifest — prerequisites (language/runtime version), the actual install command, a quick way to verify the install worked, and update/uninstall commands when the install method provides them |
| Usage | `vibe:begin:usage` / `vibe:end:usage` | Manifest scripts + real entry points — every user-facing command with realistic examples verified against the actual code |
| Documentation | `vibe:begin:docs-index` / `vibe:end:docs-index` | Every file present in `docs/` — one line per file: relative link + one-sentence description. This section is the index/summary of `docs/`: it must list ALL files in that folder, whether generated by this skill or hand-written. It is populated last, in Step 4, so it reflects the final state of `docs/` |

Cases:

- **README exists and contains managed sections:** rewrite only the inside of each marker pair. Do NOT modify a single character outside the markers — not even whitespace or heading levels.
- **README exists but has no managed sections:** do not insert them silently into a hand-written file. List the sections that could be managed and ask the user which ones to add (and where: end of file by default). Only add the pairs the user approves.
- **README does not exist:** create it with a minimal skeleton — project title, one-paragraph description (from manifest + CLAUDE.md), then the four managed sections.

Leave `vibe:begin:docs-index` / `vibe:end:docs-index` empty for now if `docs/` doesn't exist yet — it is filled in Step 4.

Mark the task `completed`.

## Step 4 — Update docs/

Mark the `Update docs/` task `in_progress`.

`docs/` files are **fully generated** — each starts with this banner:

```markdown
> Generated by /vibe:docs — manual edits will be overwritten; use README for hand-written content.
```

### Inventory the aspects to document

From the Step 1 sources (including the code walk), list every aspect a new developer must understand to work on the project. Generate **as many files as those aspects require** — no fixed catalog. Use this table as a starting point; each file is created only if its condition holds in the actual project:

| File | Aspect | Create only if |
|---|---|---|
| `docs/architecture.md` | Big picture — main modules, their responsibilities, how data flows between them (sourced from `.vibe/index.md` + `.vibe/modules/` when present, from the code otherwise) | the project has ≥ 2 identifiable modules |
| `docs/configuration.md` | Environment variables and options — name, purpose, default, required or not — derived from `.env.example` and how the code reads them | the project has configuration (env vars, config file) |
| `docs/data-model.md` | Entities, relationships, persistence | the project has a DB schema, models, or migrations |
| `docs/api.md` | HTTP endpoints or the public surface of a library — routes/exports, inputs, outputs, errors | the project exposes an API or a public library surface |
| `docs/workflows.md` | Business flows, lifecycles, state machines | the code implements multi-step processes |
| `docs/testing.md` | Test strategy — what kinds of tests exist, what each covers, how to run them | the project has a test suite |
| `docs/development.md` | Local setup, build, project-specific conventions | getting a working dev environment takes more than one obvious command |

**Open clause:** any aspect not in the table but substantial enough to confuse a new developer gets its own file — a kebab-case name describing the aspect (e.g. `docs/caching.md`, `docs/auth-flow.md`), kept stable across runs so incremental mode keeps working.

**Anti-inflation guardrails** — "as many as needed" never means "as many as possible":
- One aspect = one file. Never create an empty or near-empty file — skip a file if there is no substance for it.
- Do not paraphrase `.vibe/` or duplicate the README: each doc must give a developer-oriented reading those sources do not already provide as-is.
- If a previously generated file (banner present) documents an aspect that no longer exists in the code, delete the file and report it.

### Diagrams

When a diagram explains something better than a paragraph, embed a fenced ` ```mermaid ` block (rendered natively by GitHub):

- `graph` / `flowchart` — architecture, dependencies between modules
- `sequenceDiagram` — flows between components
- `erDiagram` — data model
- `stateDiagram-v2` — lifecycles, state machines

Rules: derive every diagram from the real code — never invent a box or an arrow; keep it small (~15 nodes max — split it otherwise); always pair it with explanatory text — the diagram illustrates, it does not replace prose.

### Rules

- **GitHub Pages site files in `docs/` are off-limits**: never modify, and never list in the README documentation index, `docs/index.html`, `docs/.nojekyll`, or any non-Markdown asset (`.html`, `.css`, `.js`, images). The index covers documentation files (Markdown) only.
- **Without `--full`**: incremental — regenerate a file only if its sources changed (compare against the current file content; if nothing meaningful changed, leave it untouched).
- **With `--full`**: regenerate every applicable file from scratch.
- If a `docs/` file exists that was NOT generated by this command (no banner): leave it alone and mention it in the report.

### Update the README documentation index

Once `docs/` reflects its final state (files created, updated, left untouched, or unchanged), rewrite the `vibe:begin:docs-index` / `vibe:end:docs-index` section in `README.md` so it lists **every** file currently in `docs/`:

- One line per file: a relative link (`[docs/architecture.md](docs/architecture.md)`) + a one-sentence, user-facing description of what it contains.
- Include hand-written files (no banner) as well as generated ones — the README must stay a complete index of `docs/`, not just of what this skill produced.
- Remove entries for files that no longer exist in `docs/`.
- If `docs/` is empty or absent, leave the section empty (or a single "No additional documentation yet." line) rather than deleting the markers.
- Keep the list sorted the same way each run (e.g. alphabetical by path) so diffs stay minimal.

Mark the task `completed`.

## Step 5 — Report

- README: which managed sections were updated / unchanged / absent, including the Documentation index
- docs/: every aspect considered, each marked created / updated / unchanged / skipped / deleted (with reason)
- Flag any `docs/` file that is not (yet) referenced in the Documentation index — this should not happen, but call it out if seen
- Do NOT print file contents unless asked
- Do NOT commit — the caller (`/vibe:feature`, `/vibe:fix`, `/vibe:release`) or the user handles commits
