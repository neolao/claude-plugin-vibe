# /vibe:sync ‚Äî Codebase Map Sync

Sync the `.vibe/` directory ‚Äî a structured context map that lets Claude understand the codebase quickly in any new session.

- **First run** (`.vibe/` absent): full generation
- **Subsequent runs**: incremental ‚Äî only modules whose source files changed since the last sync are updated
- **`--full`** in `$ARGUMENTS`: force full regeneration regardless

## Step 1 ‚Äî Read configuration

Read `CLAUDE.md` to identify source directories and stack.

## Step 2 ‚Äî Determine scope

**If `.vibe/index.md` does not exist** (or `--full` in `$ARGUMENTS`): full mode ‚Äî collect all source files, skip to Step 4.

**Otherwise ‚Äî incremental mode:**
1. Read the generation date from `.vibe/index.md` (first line: `> Generated by /vibe:sync on YYYY-MM-DD`)
2. Collect changed files from **two sources** (both are required ‚Äî `/vibe:feature` and `/vibe:fix` invoke sync *before* committing, so uncommitted changes must be detected):
   - `git log --since="YYYY-MM-DD" --name-only --pretty=format: -- <source_dirs>` ‚Äî files changed in commits since last sync
   - `git status --porcelain -- <source_dirs>` ‚Äî uncommitted files (modified, added, staged or not)
3. If **both** lists are empty: report "already up to date" and stop
4. Identify which functional zones are affected by the changed files ‚Üí only those zones will be updated

## Step 3 ‚Äî Collect source files

Gather source files for the zones to update. Exclude:
- Tests (`*.test.*`, `*.spec.*`, `test_*.py`, `tests/`, `__tests__/`)
- Dependencies (`node_modules/`, `vendor/`, `.venv/`)
- Build output (`dist/`, `build/`, `out/`, `target/`)
- Generated files (`// generated`, `# auto-generated`)
- Configuration files (`*.config.*`, `*.json` unless it contains logic)
- Migration files

## Step 4 ‚Äî Group by functional area

Infer functional zones from the directory structure (e.g. `src/auth/`, `src/api/`, `src/domain/`).
If the project is flat, group by file type or responsibility.

## Step 5 ‚Äî Write `.vibe/modules/[name].md`

For each functional zone, create one file:

```markdown
# Module: [name]
**Role:** [one sentence describing what this zone does]
**Files:** `src/auth/index.ts`, `src/auth/middleware.ts`
**Exports:** `signIn(email, password): Promise<Session>`, `AuthMiddleware`
**Depends on:** `modules/db.md`, `modules/config.md`
```

## Step 6 ‚Äî Write `.vibe/models.md`

In incremental mode: only update entries whose source files are in the changed set.

Collect all key types, interfaces, schemas, and data structures across the codebase.

```markdown
# Data models

## [ModelName]
| Field | Type | Notes |
|---|---|---|
| id | string | UUID |
| ... | ... | ... |
Defined in: `src/types/user.ts`
```

## Step 7 ‚Äî Write `.vibe/glossary.md`

Detect domain term candidates from: class names, type names, interface names, and domain-specific vocabulary **from the project's own code**.

**Exclude ‚Äî these are NOT glossary terms:**
- External library or third-party API names, methods, or call patterns (e.g. `ts_reader.run()`, `page.route()`, `cheerio.load()`)
- Browser/DOM API calls used as implementation details
- Infrastructure or protocol names that belong to a dependency, not the domain (e.g. HTTP verbs, CSS selectors, JSON keys from third-party APIs)
- Any identifier that does not map to a concept owned and defined by this project
- **Error/exception class names** ‚Äî `FooError` is a code artifact; the failure scenario it represents belongs in the definition of the relevant domain concept, not as a standalone entry
- **Implementation patterns** ‚Äî technical strategies that describe *how* the code is structured internally (e.g. "buffer-cache pattern", "repository pattern"); these are architecture notes, not domain vocabulary
- **Internal technical mechanisms** ‚Äî plumbing details that exist purely to enable an implementation and have no meaning outside the code (e.g. synthetic/fake URLs used as cache keys)
- **Configuration field names or values** ‚Äî a YAML key or its allowed values (e.g. `mode: fast | safe`) is a config API detail, not a domain concept with independent meaning

A valid glossary term is a **business concept** the project owns and names ‚Äî something that defines *what the domain does*, not *how the code achieves it*. Ask: would a domain expert (someone who knows the project's business domain but not this codebase) recognize this term as meaningful? If not, it does not belong here.

If `.vibe/glossary.md` already exists:
1. **Audit existing terms** against the exclusion criteria above. For each term that violates them, flag it as a candidate for removal.
2. If any candidates are found, **list them to the user** with a one-line reason for each, and ask for confirmation before removing them. Do not remove anything without explicit approval.
3. **Preserve all approved definitions**, only add newly detected terms.

New terms are added with a minimal entry ‚Äî definitions are to be refined by the user:

```markdown
# Ubiquitous Language

## [Domain term]
**Definition:** [what this concept means in the domain ‚Äî fill in]
**Code:** `ClassName`, `functionName()` in `src/...`
**Do not confuse with:** [similar but distinct term, if applicable]
```

## Step 8 ‚Äî Write `.vibe/index.md`

**Compactness constraint:** `index.md` is imported into every session via `@.vibe/index.md` in CLAUDE.md ‚Äî its token cost is paid at every session start. Keep it minimal: one line per module, a short list of observed patterns, no detailed content. Details belong in `modules/`, `models.md`, and `glossary.md`, which are read on demand.

```markdown
# [PROJECT_NAME] ‚Äî Codebase index
> Generated by /vibe:sync on YYYY-MM-DD. Re-run to update.

## Modules
- [`modules/auth.md`](modules/auth.md) ‚Äî [one-line description]
- [`modules/api.md`](modules/api.md) ‚Äî [one-line description]
...

## Observed patterns
[Patterns and conventions actually found in the code ‚Äî not those declared in CLAUDE.md]

## Other context files
- [`models.md`](models.md) ‚Äî data models
- [`glossary.md`](glossary.md) ‚Äî ubiquitous language
- [`decisions/`](decisions/) ‚Äî architectural decision records (if the directory exists)
```

Note: `.vibe/decisions/` is maintained by `/vibe:feature` ‚Äî not auto-generated by sync.

## Step 8b ‚Äî Write `.vibe/README.md`

`.vibe/` mixes regenerable artifacts with irreplaceable data ‚Äî this README tells humans and agents which is which.

- **Full mode:** (re)write the file.
- **Incremental mode:** create it only if absent (the content is static).

```markdown
# .vibe/ ‚Äî Project context

Maintained by the `/vibe:*` commands. Lifecycle of each entry:

| Entry | Lifecycle |
|---|---|
| `README.md`, `index.md`, `modules/`, `models.md` | ‚ôª Regenerable ‚Äî recreated by `/vibe:sync`, do not edit by hand |
| `glossary.md` | ‚öÝ Hybrid ‚Äî structure generated, **definitions refined by hand**: never delete without review |
| `decisions/` | üîí ADRs ‚Äî append-only, never regenerated, do not delete |
| `backlog/` | üîí Work data ‚Äî never regenerated, do not delete |

Never delete the whole `.vibe/` directory: only the ‚ôª entries can be recovered by re-running `/vibe:sync`.
```

**Legacy migration:** if an old single-file `.vibe/decisions.md` exists, propose to the user (explicit confirmation required) to split it: each `## [YYYY-MM-DD] Title` section becomes `decisions/NNN-slug.md` (numbered in chronological order, `date:` taken from the section heading, `status: accepted`), then delete the legacy file. If the user declines: leave it untouched and add a `decisions.md` row to the README table marked `üîí Legacy ‚Äî superseded by decisions/`.

## Step 9 ‚Äî Report

- N modules updated (out of N total)
- N data models updated
- N terms in glossary (N new, N preserved)
- Do NOT print file contents unless asked
