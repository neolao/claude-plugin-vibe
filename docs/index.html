why vibe

Quality is automated, not hoped for

Every guarantee a careful team enforces by discipline, the plugin enforces by workflow — on any project it's installed on.

01

TDD-first, always

Features and fixes start with a failing test, written in your stead — and every test is checked against a shared definition of a tautological test, so green means something.

02

Proven, not just green

After implementation the change is exercised for real — nominal path plus an edge case — instead of trusting the suite. If the app won't even launch, it records a working launch recipe and retries.

03

Twelve-angle review

One command fans out specialized agents — security, complexity, naming, robustness, tests, and more — in parallel, each auditing a single dimension, then fixes what they find.

04

A backlog that commits itself

Queue ideas as numbered items, singly or in bulk; bundled requests are split into independent items automatically. Implement any of them later just by number.

05

Feedback loops that close

Pre-existing test failures are offered for tracking, a reminder fires when several changes ship unreviewed, and every dead end is logged with its diagnosis so no session rediscovers it.

06

Nothing silently lost

Work is committed before every turn ends — even interruptions — and a one-command release finalizes the changelog, refreshes docs, bumps the version, and tags.

skills in action

Watch a session, not a promise

Each demo below replays a scripted /vibe:* session, exactly as the workflow behaves.

/vibe:feature — TDD, then proof

Pick a backlog item by number. The test comes first and fails for the right reason; only then is the feature implemented. Green tests aren't the finish line — the change is exercised end-to-end before the commit, and the changelog is updated on the way out.

claude — /vibe:feature

/vibe:review — twelve auditors at once

Review agents run in parallel, each on a single quality dimension, with a color-coded status line. The test-review agent doesn't just read your tests — it runs the real suite and flags tests that can't actually fail. Findings are fixed, not filed away.

claude — /vibe:review

/vibe:backlog — ideas, safely queued

Toss in a request; if it bundles several independent capabilities it's split automatically into separate numbered items, each committed on the spot. List them anytime — and get nudged when too much has shipped without a review.

claude — /vibe:backlog

/vibe:release — a version in one command

Changelog finalized under the new version, docs refreshed, version bumped, commit and tag created. The bump can be explicit (1.3.0) or inferred from the unreleased entries (major / minor / patch).

claude — /vibe:release

typical flow

From empty repo to tagged release

  1. /vibe:init — once, to set up CLAUDE.md and the codebase context map. Both are committed with your project, so every session and every teammate starts from the same conventions.
  2. /vibe:backlog "some feature idea" — queue work as it comes to mind.
  3. /vibe:feature 001 — or /vibe:fix "bug description" — implement, TDD-first, proven end-to-end.
  4. /vibe:review — periodically, to catch quality issues across twelve dimensions.
  5. /vibe:release patch|minor|major — ship a version.
