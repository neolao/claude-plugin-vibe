# Development

How to work on the plugin itself. There is nothing to install or build: the deliverables are Markdown instruction files, one shell script, and three JSON manifests, all consumed directly by Claude Code.

## No tests, no linter — by design

This repo deliberately has no test suite, package manifest, or linter: there is no executable application code, only definitions interpreted by Claude Code. Quality relies on:

- re-reading generated content before presenting it,
- strict structural consistency with neighboring files (same frontmatter, tone, level of detail),
- syntactic validation of touched JSON manifests (`jq . .claude-plugin/plugin.json`, etc.).

Do not introduce a `package.json`, test framework, or linter without an explicit request.

## Adding or changing a skill

1. Find the closest existing skill in `skills/*/SKILL.md` and mirror its structure.
2. A skill lives in `skills/<name>/SKILL.md` with frontmatter:
   - `name` — slug matching the directory, invoked as `/vibe:<name>`
   - `description` — one line, shown in skill listings
   - `argument-hint` — optional placeholder for `$ARGUMENTS`
   - `user-invocable` — optional, default `true`; set `false` for an internal skill hidden from the `/` menu and callable only by other skills via the Skill tool (see `skills/tasks/SKILL.md`)
3. If the change is user-visible, update `README.md` (the "Skills (commands)" table is hand-written; the Features/Install/Usage/Documentation sections between `vibe:begin:*`/`vibe:end:*` markers are regenerated by `/vibe:docs` — never edit inside markers by hand).
4. Record the change in `CHANGELOG.md` under `[Unreleased]` (Keep a Changelog format, end-user voice).

## Adding or changing a review agent

Same approach: one file `agents/review-<dimension>.md` with frontmatter `name` + `description`, auditing exactly **one** quality dimension. If a check borders an existing agent's territory, add an explicit delegation note in both files — every check must have a single owning agent. Conditionally-relevant agents (e.g. `review-hexagonal`, `review-pentest`) state their activation condition in the description.

## Conventions that must hold

- No secrets or credentials in any file.
- No orphan skill/agent files after a rename or removal.
- JSON manifests (`plugin.json`, `marketplace.json`, `settings.json`) stay syntactically valid.
- The `.vibe/` map of this repo is dogfooding — regenerate it with `/vibe:sync` rather than editing it by hand (entry lifecycles are listed in `.vibe/README.md`).

## Releasing

Run `/vibe:release patch|minor|major` (or an explicit version): it finalizes the changelog, refreshes the docs, bumps `version` in `.claude-plugin/plugin.json`, commits and tags. Install locally for testing via `/plugin marketplace add neolao/claude-plugin-vibe` + `/plugin install vibe`.
